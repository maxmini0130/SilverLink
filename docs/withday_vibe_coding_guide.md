# SilverLink Vibe Coding Guide

## 1. 문서 목적

이 문서는 SilverLink를 Claude, Codex 등 AI 코딩 도구로 안정적으로 개발하기 위한 기준 문서다.

이 문서의 목적은 다음과 같다.

1. 서비스의 핵심 방향을 흔들리지 않게 유지한다
2. 불필요한 기능 확장을 막는다
3. AI가 한 번에 너무 많은 파일을 만들거나 구조를 복잡하게 하지 않도록 한다
4. 구현 우선순위를 명확히 한다
5. 현재 단계에서 무엇을 만들고, 무엇을 만들지 말아야 하는지 분명히 한다

---

## 2. 서비스 핵심 정의

SilverLink는 **중장년·시니어가 부담 없이 좋은 인연을 만나고, 대화와 동행으로 자연스럽게 관계를 이어가는 안심형 인연·활동 서비스**다.

중요:
- 이 서비스의 본질은 “좋은 사람과 연결되고 싶은 욕구”다
- 다만 노골적인 데이팅앱처럼 보이면 안 된다
- 복지앱, 건강앱, 종합생활앱처럼 퍼지면 안 된다
- 핵심은 인연, 동행, 신뢰, 안전, 활동 연결이다

---

## 3. 절대 흔들리면 안 되는 제품 방향

### 3.1 서비스 본질
- 좋은 인연을 시작하는 서비스
- 대놓고 소개팅은 아니지만, 핵심은 사람과의 연결이다
- 외로움, 대화 욕구, 동행 욕구, 이성에 대한 관심을 자연스럽게 다룬다

### 3.2 서비스 얼굴
- 안심형
- 점잖은
- 생활형
- 부담 없는
- 품위 있는

### 3.3 서비스가 아니어야 하는 것
- 건강관리 앱
- 복약 알림 앱
- 운세 앱
- 산책 미션 앱
- 종합 커뮤니티 앱
- 공공복지 정보 앱
- 너무 노골적인 시니어 데이팅앱

## 4. 현재 단계에서 반드시 유지해야 할 핵심 구조

### 4.1 핵심 사용자 흐름
#### 흐름 A
회원가입 → 온보딩 → 홈 → 사람 상세 → 관심 보내기 → 대화

#### 흐름 B
회원가입 → 온보딩 → 홈 → 피드 보기 → 반응 → 프로필 → 관계 연결

#### 흐름 C
회원가입 → 온보딩 → 홈 → 모임 → 참여 → 모임 채팅 → 후속 관계

### 4.2 관계 구조
- 관심: 가벼운 신호
- 연결(또는 1촌): 상호 승인된 관계
- 대화: 상호 관심, 연결, 같은 모임 참여 시 가능

### 4.3 홈 우선순위
1. 추천 사람
2. 최근 공개 피드
3. 추천 모임
4. 알림 요약

## 6. 지금 만들면 안 되는 것

현재 단계에서 아래 기능은 만들지 않는다.

- 운세
- 복약 알림
- 건강 체크
- 산책 미션
- 습관 관리
- 종합 생활편의 기능
- 결제
- 프리미엄 구독
- 광고 시스템
- 보호자 기능
- 여행 상품화
- 고급 추천 알고리즘
- 분석 대시보드
- 대규모 관리자 시스템

이유:
핵심 욕구를 흐리게 하고, MVP의 선명도를 떨어뜨리기 때문이다.

## 7. 디자인 방향

### 7.1 한 줄 정의
따뜻하고 정돈된 안심형 생활 관계 플랫폼

### 7.2 디자인 원칙
- 시니어 친화 UX
- 큰 글자
- 큰 버튼
- 카드 중심 구조
- 댓글 없는 구조
- 과도한 장식 금지
- 너무 복지앱처럼 보이지 않기
- 너무 소개팅앱처럼 보이지 않기

### 7.3 컬러 방향
- 메인: 차분한 틸 / 블루그린
- 보조: 따뜻한 살구 / 코랄
- 배경: 부드러운 웜톤 라이트

### 7.4 홈 화면 인상
- 기능 메뉴판처럼 보이면 안 됨
- 추천 중심
- 첫 행동이 보여야 함

## 8. AI 코딩 작업 원칙

### 8.1 반드시 지켜야 할 원칙
1. 한 번에 너무 많은 파일을 만들지 말 것
2. 항상 먼저 수정 대상 파일 구조를 보여줄 것
3. 기능 로직보다 먼저 서비스 방향을 이해할 것
4. 기존 로직을 불필요하게 깨지 말 것
5. 대규모 리팩터링 금지
6. 모바일 우선으로 구현할 것
7. 시니어 UX를 계속 유지할 것
8. MVP 범위를 벗어나는 제안은 최소화할 것

### 8.2 코드 작성 방식
- 최소 수정 원칙
- 파일 단위로 점진 수정
- 먼저 계획, 그 다음 코드
- 완료 후 어떤 체크리스트 항목이 해결됐는지 설명
- 다음 작업은 항상 1개만 제안

### 8.3 금지 사항
- 추상화 과다
- 폴더 구조 전면 재편
- 컴포넌트 대량 생성
- 새 기능 욕심
- 제품 방향과 상관없는 부가 기능 추가

## 10. AI에게 먼저 시킬 일

AI에게는 아래 순서로 시키는 것이 좋다.

### Step 1
현재 프로젝트 구조 분석

### Step 2
출시 전 체크리스트 기준 점검

### Step 3
가장 먼저 수정할 항목 1개 선정

### Step 4
해당 항목만 최소 범위 수정

### Step 5
수정 후 다시 체크리스트 기준 점검

## 11. 추천 프롬프트 예시

### 11.1 프로젝트 점검용
현재 프로젝트를 기준으로 출시 전 기능 체크리스트 점검 작업을 진행하자.

참고 문서:
- docs/silverlink_mvp.md
- docs/silverlink_prd.md
- docs/silverlink_design_guide.md
- docs/priority_plan.md
- docs/release_checklist.md
- docs/silverlink_vibe_coding_guide.md

전제:
- 지금 단계의 목표는 신규 기능 추가가 아니라 안정화, 핵심 사용자 흐름 점검, 관계 모델 정리, UX 정리다.
- 기존 기능 로직을 크게 흔들지 말고, 무엇이 준비되어 있고 무엇이 불안정한지 먼저 파악해야 한다.

이번 단계에서 할 일:
1. release_checklist.md 기준으로 현재 코드베이스를 점검해라
2. 체크리스트 항목 중 이미 충족된 것 / 불안정한 것 / 누락된 것을 구분해라
3. 가장 먼저 수정해야 할 항목 5개를 우선순위 순으로 정리해라
4. 각 항목마다 관련 파일 위치를 추정해라
5. 마지막에 “지금 바로 수정할 첫 작업 1개”만 추천해라

중요:
- 신규 기능 제안 최소화
- 기존 구현을 유지하는 방향 우선
- 대규모 리팩터링 금지
- 한국어로 작성

### 11.2 단일 작업 수정용
좋아. 이제 우선순위 1번 항목만 수정하자.

원칙:
- 한 번에 하나만 고칠 것
- 파일 구조 먼저 보여줄 것
- 수정 대상 파일만 최소 범위로 건드릴 것
- 기존 기능을 깨지 않게 할 것
- 수정 후 어떤 체크리스트 항목이 해결되는지 설명할 것

먼저 수정 계획부터 보여줘.

## 12. 최종 정리

SilverLink는 “시니어 종합생활앱”이 아니다.

SilverLink는
- 좋은 인연을 시작하고
- 생활 피드로 신뢰를 만들고
- 관심과 연결을 통해 대화를 열고
- 모임과 동행으로 관계를 이어가는
안심형 인연·활동 서비스다.

따라서 모든 구현 판단 기준은 아래 한 문장을 기준으로 삼는다.

**이 기능이 좋은 인연을 더 자연스럽고 안전하게 시작하게 만드는가?**

이 질문에 답이 “아니오”라면 지금 단계에서는 만들지 않는다.
