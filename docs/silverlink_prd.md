# SilverLink MVP PRD

## 1. 문서 정보
- 문서명: SilverLink MVP PRD
- 서비스명: SilverLink (실버링크)
- 문서 목적: Claude 및 개발 착수용 기준 문서
- 문서 버전: v0.1
- 기준 범위: MVP

---

## 2. 제품 개요

### 2.1 한 줄 정의
시니어가 일상 사진과 취미를 공유하고, 공개 범위를 직접 조절하며, 비슷한 사람과 안전하게 연결되는 미니홈피형 관계 서비스

### 2.2 제품 성격
- 노골적인 소개팅 앱이 아님
- 댓글 중심 SNS가 아님
- 외부 웹에 완전 공개되는 공개형 SNS가 아님
- 친구, 말벗, 동행, 취미 기반 관계가 자연스럽게 이어지는 구조를 지향
- 싸이월드식 공개범위 감성과 인스타식 피드 일부를 결합한 구조

### 2.3 해결하려는 문제
1. 시니어가 자연스럽게 새로운 사람을 만날 수 있는 디지털 공간이 부족함
2. 노골적인 소개팅 서비스는 거부감이 있음
3. 공개형 SNS는 과도하게 열려 있거나 피로도가 높음
4. 시니어 친화 UI와 안전한 관계 형성 구조가 부족함
5. 온라인 관계가 실제 활동이나 모임으로 이어지기 어려움

---

## 3. 목표와 비목표

### 3.1 제품 목표
- 시니어가 부담 없이 프로필과 일상을 공유할 수 있게 한다
- 자연스러운 관계 형성을 유도한다
- 공개범위와 관계 단계를 기반으로 안전한 소통 구조를 만든다
- 온라인 연결을 모임으로 이어지게 한다
- MVP 기준으로 핵심 사용자 흐름을 빠르게 검증한다

### 3.2 비목표
- 초기부터 대규모 공개형 SNS를 만드는 것
- 초기부터 강한 연애/재혼 서비스로 포지셔닝하는 것
- 초기부터 결제/광고/프리미엄 수익화를 붙이는 것
- 초기부터 영상, 음성, AI 추천, 보호자 기능까지 모두 넣는 것

## 4. 핵심 사용자

### 4.1 주요 타깃
- 50대 후반 ~ 70대 초반
- 스마트폰 사용은 가능하나 복잡한 서비스는 어려울 수 있음
- 친구, 말벗, 동행, 취미 기반 관계를 원하는 사용자
- 노골적인 소개팅보다는 자연스러운 관계 형성을 선호하는 사용자

### 4.2 사용자 특성
- 사진, 피드, 좋아요, DM 개념에 점점 익숙해지는 세대
- 공개 범위와 안전에 민감함
- 댓글, 비교, 과도한 노출보다는 통제 가능한 관계 구조를 선호할 가능성이 높음

### 4.3 대표 페르소나
#### 페르소나 A
- 62세 여성
- 서울 거주
- 산책, 전시, 카페를 좋아함
- 편하게 대화하고 동네에서 만날 수 있는 사람을 찾고 싶음
- 노골적인 소개팅 앱은 부담스러움

#### 페르소나 B
- 67세 남성
- 경기 거주
- 사진, 여행, 가벼운 모임을 좋아함
- 비슷한 취미를 가진 사람과 자연스럽게 친해지고 싶음
- 앱 사용은 가능하지만 복잡한 설정은 어려워함

## 5. 핵심 가치 제안
1. 안전한 관계 형성
2. 시니어 친화 UX
3. 공개범위를 직접 조절할 수 있는 생활 피드

## 8. 공개범위 정책

### 8.1 공개범위 옵션
- 나만 보기
- 1촌만 보기
- 관심 있는 사람만 보기
- 같은 모임 사람만 보기
- 전체 인증회원 보기

### 8.2 정책 원칙
- 외부 웹 검색 노출 없음
- 인증 회원 중심의 제한형 커뮤니티
- 공개범위는 DB 접근 정책으로 제어
- 차단 사용자 간에는 조회 및 대화 제한
- DM 가능 여부는 화면 노출과 서버 정책이 동시에 일치해야 함

## 9. 핵심 사용자 시나리오

### 시나리오 1: 사람 중심 관계 형성
1. 사용자가 회원가입한다
2. 프로필을 작성한다
3. 홈에서 추천 사람을 본다
4. 프로필 상세를 확인한다
5. 관심을 보낸다
6. 상호 관심이 성사되면 대화가 가능해진다

### 시나리오 2: 피드 중심 관계 형성
1. 사용자가 홈에서 새 피드를 본다
2. 반응을 보낸다
3. 상대 프로필을 확인한다
4. 관심을 보낸다
5. 조건 충족 시 대화로 이어진다

### 시나리오 3: 모임 중심 관계 형성
1. 사용자가 모임을 탐색한다
2. 모임 상세를 본다
3. 모임에 참여한다
4. 모임 대화방에 들어간다
5. 오프라인 모임에 참여한다
6. 이후 1촌 또는 대화로 이어진다

## 11. 핵심 화면 정의

### 11.1 홈
#### 목적
앱에 들어오자마자 오늘 할 행동을 추천받는 메인 허브

#### 주요 UI 요소
- 오늘의 추천 사람 카드
- 주변 새 피드 카드
- 추천 모임 카드
- 알림 요약

#### 주요 액션
- 프로필 보기
- 관심 보내기
- 피드 반응
- 모임 참여

#### 주요 상태
- 신규 사용자 빈 상태
- 추천 데이터 있음
- 알림 있음
- 알림 없음

#### 제한 조건
- 과도한 정보량 금지
- 첫 화면에서 복잡한 입력 없음

### 11.2 사람 상세 프로필
#### 목적
상대를 신뢰하고 관계 시작 여부를 판단

#### 주요 UI 요소
- 대표 사진
- 이름/닉네임
- 나이대/지역
- 관심사
- 관계 목적
- 자기소개
- 생활 피드 미리보기
- 관심 보내기 / 1촌 신청 / 대화 버튼

#### 주요 상태
- 관계 없음
- 내가 관심 보냄
- 상호 관심
- 1촌
- 같은 모임 참여자

#### 제한 조건
- 차단 상태 시 접근 제한
- 대화 가능 상태가 아니면 DM 버튼 비활성 또는 안내

### 11.3 피드 등록 / 피드 상세
#### 목적
일상과 취미를 공유하고 관계 포인트를 형성

#### 주요 UI 요소
- 사진 업로드
- 짧은 글
- 공개범위 선택
- 반응 버튼
- 댓글 없음

#### 주요 상태
- 공개범위별 게시 상태
- 반응 허용/비허용
- 피드 없음
- 피드 있음

#### 제한 조건
- 댓글 기능 없음
- 접근 권한 없는 사용자는 피드 조회 불가

### 11.4 모임 목록 / 모임 상세

- 관심사 필터
- 모임 카드
- 모임 설명

- 참여완료
- 마감
- 종료

#### 제한 조건

## 12. 기능 요구사항

### 12.1 계정 및 인증
- 사용자는 계정을 생성할 수 있어야 한다
- 로그인 후 세션이 유지되어야 한다
- 사용자 인증 상태를 구분할 수 있어야 한다

### 12.2 프로필
- 사용자는 프로필을 작성/수정할 수 있어야 한다
- 프로필에는 관심사와 관계 목적이 포함되어야 한다
- 프로필 사진을 등록할 수 있어야 한다

### 12.3 피드
- 사용자는 피드를 생성/수정/삭제할 수 있어야 한다
- 피드에는 사진과 글이 포함될 수 있어야 한다
- 각 피드는 공개범위를 가져야 한다
- 댓글 없이 반응만 가능해야 한다

### 12.4 사람 탐색
- 사용자는 추천 사람을 볼 수 있어야 한다
- 지역/관심사 기반으로 사람을 찾을 수 있어야 한다
- 프로필 상세 페이지로 진입할 수 있어야 한다

### 12.5 관계 연결
- 사용자는 관심을 보낼 수 있어야 한다
- 상호 관심 여부를 판단할 수 있어야 한다
- 1촌 관계를 관리할 수 있어야 한다

### 12.6 대화
- 허용된 조건에서만 대화가 가능해야 한다
- 개인 대화 목록을 확인할 수 있어야 한다
- 대화를 차단/신고할 수 있어야 한다

### 12.7 모임
- 사용자는 모임 목록을 볼 수 있어야 한다
- 모임 상세를 볼 수 있어야 한다
- 모임에 참여할 수 있어야 한다

### 12.8 안전
- 사용자는 상대를 차단할 수 있어야 한다
- 사용자는 신고를 제출할 수 있어야 한다
- 차단 관계에서는 조회 및 대화가 제한되어야 한다

## 13. 비기능 요구사항

### UX
- 시니어 친화적이어야 함
- 큰 글씨와 큰 버튼을 사용해야 함
- 선택지를 줄이고 복잡한 인터랙션을 피해야 함

### 성능
- 홈, 프로필, 모임 목록 등 주요 화면은 빠르게 로드되어야 함
- 이미지 로딩이 과도하게 느리지 않아야 함

### 보안
- 인증 기반 접근 제어가 필요함
- 모든 핵심 데이터는 공개범위 및 권한 정책을 따라야 함
- 차단/신고 관련 데이터는 일반 사용자에게 노출되지 않아야 함

### 데이터 정책
- 외부 완전 공개 금지
- 공개범위는 UI가 아니라 데이터 접근 정책으로도 보장되어야 함

## 16. 기술 아키텍처 방향

### MVP 아키텍처
- Next.js가 프론트엔드 및 기본 서버 레이어 역할
- Supabase가 Auth / Postgres / Storage 담당
- 공개범위 및 권한은 RLS 중심으로 제어
- 중요한 액션은 Next.js Server Action 또는 Route Handler로 처리

### 원칙
- 단순 조회는 비교적 빠르게 구현
- 관계 생성, 대화 가능 여부, 신고/차단 등 핵심 로직은 서버 레이어로 점진 이동
- 프론트에 비즈니스 규칙을 과하게 넣지 않음

## 19. 스택에 대한 실무 판단

### 그대로 가져가도 되는 부분
- Next.js
- TypeScript
- Tailwind CSS
- shadcn/ui
- React Hook Form
- Zod
- Zustand
- Vercel

### Supabase 판단
Supabase는 MVP용 임시 선택이 아니라 초기 운영 단계까지 충분히 사용할 수 있다.
다만 아래 원칙이 중요하다.
- RLS를 초반부터 제대로 설계
- 공개범위/권한을 DB 정책으로 제어
- 프론트가 DB를 무분별하게 직접 다루지 않도록 주의
- 관계/대화/신고 같은 핵심 로직은 서버 레이어로 점진 이동

### Prisma 판단
초기에는 굳이 넣지 않는다.
복잡한 서버 로직, 백오피스, 배치 작업, 운영툴 필요성이 커질 때 검토한다.

## 20. 오픈 이슈
- 초기 인증 수준을 어디까지 둘 것인가
- 추천 로직을 MVP에서 얼마나 단순하게 둘 것인가
- 모임 생성 권한을 모든 사용자에게 열 것인가
- 1촌과 상호 관심의 관계를 어떻게 정리할 것인가
- 피드 공개범위 UX를 얼마나 단순하게 보여줄 것인가

구조로 설계한다.
